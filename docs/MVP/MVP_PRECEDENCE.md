# Dorea Network — MVP Precedence Rules

## Purpose

This document establishes implementation authority and execution precedence for the Dorea MVP.

The Dorea repository contains both:

1. Long-range architectural doctrine
2. Near-term MVP execution guidance

Both are intentional.

However:

During MVP development and pilot validation, execution clarity and operational simplicity must take precedence over architectural completeness.

This document exists to prevent:
- scope creep
- premature platformization
- architectural overbuilding
- governance overengineering
- unnecessary runtime complexity

before the core mission is operationally validated.

---

# Core Principle

> The MVP exists to validate whether Dorea meaningfully helps ensure that no one is left alone.

All implementation decisions during MVP development should optimize for:
- emotional usefulness
- operational reliability
- pastoral trust
- volunteer sustainability
- human follow-through
- low-friction engagement

NOT:
- architectural elegance
- generalized platform capability
- federation readiness
- theoretical extensibility
- runtime sophistication

---

# Precedence Hierarchy

During MVP development, documents located in:

```text
/docs/mvp/
```

take implementation precedence over:
- long-range architectural doctrine
- future-state runtime models
- advanced governance concepts
- speculative scalability systems
- future federation capabilities

if conflicts or ambiguity arise.

---

# Long-Range Doctrine Status

Documents outside `/docs/mvp/` remain intentional and valuable.

They serve as:
- architectural direction
- platform philosophy
- governance doctrine
- continuity principles
- future-state vision
- system integrity guidance

However:

They are NOT automatically mandatory for MVP implementation.

Their existence does NOT imply:
- immediate implementation
- MVP necessity
- pilot requirement
- operational readiness

---

# MVP Override Rule

If a conflict exists between:

- a long-range doctrine document
and
- an MVP implementation document

the MVP document wins until:
- the pilot phase is complete
- MVP validation criteria are met
- operational usefulness is proven
- the platform formally enters post-MVP expansion

---

# MVP Success Before Expansion

No major architectural expansion should occur until the MVP successfully demonstrates:

- requests are acknowledged reliably
- follow-up continuity works consistently
- unresolved needs remain visible
- pastors trust the system
- volunteers remain engaged
- members feel supported
- pilot communities experience measurable operational value

---

# Explicitly Deferred Concepts

The following concepts are intentionally deferred during MVP execution unless explicitly re-approved:

- federation between communities
- advanced replay engines
- deep telemetry analytics
- policy simulation systems
- generalized runtime orchestration
- broad agent ecosystems
- advanced adaptive governance
- complex command center analytics
- multi-community synchronization
- advanced recommendation systems
- high-scale optimization architecture

These concepts may remain documented as future-state architecture but are not implementation priorities.

---

# Simplicity Rule

When implementation decisions are unclear:

Choose the solution that:
- reduces operational burden
- simplifies onboarding
- minimizes moderator workload
- improves response reliability
- increases emotional trust
- accelerates pilot learning
- preserves human-centered interaction

---

# Human-Centered Rule

Dorea MVP is fundamentally:
- a continuity system
- a care coordination platform
- a human support network

The MVP is NOT attempting to prove:
- autonomous ministry systems
- AI-driven church governance
- generalized social infrastructure
- large-scale agent orchestration

Human ownership remains central.

---

# Pilot-First Rule

The MVP must optimize for:
- learning
- operational clarity
- emotional usefulness
- real-world ministry viability

NOT:
- platform completeness
- technical novelty
- future scalability assumptions
- generalized ecosystem readiness

---

# Repository Philosophy

The Dorea repository intentionally contains architecture beyond the MVP.

This is acceptable and intentional.

The repository represents:
- vision
- doctrine
- future-state direction
- platform continuity

The MVP implementation represents:
- proof
- validation
- operational learning
- emotional usefulness
- ministry viability

Both layers are important.

However:

Until MVP validation is complete, execution discipline takes precedence over architectural expansion.

---

# Final Principle

If uncertainty exists during MVP implementation, ask:

> Does this directly help ensure that no one is left alone during the pilot experience?

If not:
- simplify it
- defer it
- or remove it from MVP scope.
