# Dorea Network — MVP Acceptance Criteria

## Purpose

This document defines the minimum operational acceptance criteria for the Dorea MVP pilot.

The purpose of these criteria is to determine whether the MVP successfully validates the core Dorea mission:

> No one is left alone.

These criteria intentionally prioritize:
- continuity
- acknowledgment
- follow-through
- emotional usefulness
- operational reliability
- pilot sustainability

The MVP is NOT evaluated based on:
- platform sophistication
- architectural completeness
- analytics depth
- federation readiness
- advanced AI capability

---

# MVP Validation Philosophy

The MVP succeeds if it meaningfully improves:

- acknowledgment consistency
- visibility of unresolved needs
- follow-up continuity
- pastoral awareness
- volunteer coordination
- member perception of support

The MVP fails if:
- requests disappear silently
- volunteers become overwhelmed
- pastors lose trust
- operational burden becomes excessive
- users feel emotionally unsafe
- the system becomes administratively heavy

---

# Core MVP Acceptance Criteria

## AC-1 — Every Request Is Acknowledged

Every submitted request must receive acknowledgment.

### Success Criteria

- acknowledgment delivered within:
  ```text
  60 seconds
  ```

- acknowledgment status visible to requester
- acknowledgment event recorded

### Failure Conditions

- requests remain unacknowledged
- acknowledgment visibility unclear
- acknowledgment failures occur silently

---

# AC-2 — Every Request Has Human Visibility

Every request must become visible to at least one responsible human.

### Success Criteria

- all requests appear in operational queues
- no hidden unresolved requests
- assignment visibility exists
- unresolved requests remain surfaced

### Failure Conditions

- requests become orphaned
- requests disappear from visibility
- no accountable owner exists

---

# AC-3 — Human Ownership Exists

Every active request must support assignment to a human owner.

### Success Criteria

- ownership status visible
- assignment changes tracked
- reassignment supported
- assigned volunteers/leaders notified

### Failure Conditions

- unclear ownership
- assignment ambiguity
- unresolved unassigned requests

---

# AC-4 — Follow-Up Continuity Works

The system must support ongoing follow-up until intentional resolution.

### Success Criteria

- follow-up actions recorded
- reminders generated
- unresolved requests remain visible
- stale requests surfaced automatically

### Failure Conditions

- requests silently expire
- follow-up gaps remain hidden
- no continuity tracking exists

---

# AC-5 — Escalation Handling Works

Urgent concerns must support human escalation handling.

### Success Criteria

- escalation requests route to authorized leaders
- escalation visibility restricted appropriately
- escalation actions logged
- leaders notified successfully

### Failure Conditions

- escalations remain unseen
- improper visibility exposure
- no escalation ownership exists

---

# AC-6 — Mobile Experience Is Usable

Core workflows must function reliably on mobile devices.

### Success Criteria

Members can complete:
- request submission
- acknowledgment review
- follow-up review
- notifications
- basic communication

using mobile-first interfaces.

### Failure Conditions

- excessive navigation complexity
- desktop dependency
- poor mobile responsiveness
- high-friction interaction

---

# AC-7 — Volunteer Workflow Is Sustainable

Volunteers must be able to operate the system without burnout or excessive operational burden.

### Success Criteria

- volunteers understand responsibilities
- workflows require minimal training
- assignment management remains simple
- volunteer retention remains stable during pilot

### Failure Conditions

- operational overload
- confusing workflows
- excessive administrative burden
- rapid volunteer disengagement

---

# AC-8 — Pastoral Visibility Improves

Pastors and leaders must gain meaningful visibility into unresolved needs.

### Success Criteria

Leaders can clearly identify:
- unresolved requests
- aging requests
- escalation requests
- assignment gaps
- follow-up delays

### Failure Conditions

- unresolved needs remain hidden
- leaders lack operational visibility
- escalation blind spots exist

---

# AC-9 — Emotional Safety Is Maintained

Members must feel emotionally safe using the platform.

### Success Criteria

- users understand visibility boundaries
- privacy expectations are clear
- support interactions feel compassionate
- users report feeling acknowledged

### Failure Conditions

- users fear exposure
- confidentiality confusion exists
- interactions feel cold or bureaucratic
- members hesitate to request help

---

# AC-10 — Operational Simplicity Is Preserved

The MVP must remain operationally lightweight.

### Success Criteria

- minimal onboarding friction
- low moderation burden
- limited governance overhead
- understandable workflows

### Failure Conditions

- excessive configuration
- operational complexity growth
- enterprise-style administration
- unclear governance processes

---

# Pilot Operational Targets

## Acknowledgment Target

```text
< 60 seconds
```

---

## Human Assignment Target

```text
< 4 hours
```

---

## Escalation Review Target

```text
< 15 minutes
```

for urgent escalation events.

---

## Follow-Up Cadence Target

```text
At least one visible follow-up every 24 hours
```

until:
- resolved
- escalated
- reassigned

---

# MVP Success Indicators

The pilot is considered successful if:

- >95% acknowledgment reliability achieved
- unresolved requests remain visible
- pastors report improved continuity visibility
- volunteers remain operationally sustainable
- members report feeling supported
- operational overhead remains manageable
- escalation handling remains clear and reliable

---

# MVP Failure Indicators

The MVP should be reconsidered if:

- requests disappear silently
- acknowledgment reliability fails repeatedly
- volunteers become overwhelmed
- leaders lose operational trust
- escalation handling becomes inconsistent
- onboarding complexity blocks adoption
- users stop engaging due to friction or emotional discomfort

---

# Pilot Feedback Requirements

The pilot should gather feedback from:

## Members

Questions:
- Did you feel acknowledged?
- Did follow-up feel genuine?
- Did the experience feel emotionally safe?
- Did requesting help feel easy?

---

## Volunteers

Questions:
- Was the workload manageable?
- Were requests clear?
- Did escalation rules feel understandable?
- Was follow-up sustainable?

---

## Pastors / Leaders

Questions:
- Did visibility improve?
- Were unresolved needs easier to track?
- Did the platform reduce ministry blind spots?
- Was operational overhead acceptable?

---

# Deferred Acceptance Areas

The MVP intentionally does NOT evaluate:

- federation scalability
- advanced analytics
- replay intelligence
- generalized orchestration
- adaptive governance
- AI optimization quality
- engagement maximization
- social growth mechanics

These are post-MVP concerns.

---

# Acceptance Governance Rule

If an implementation decision improves:
- reliability
- simplicity
- continuity
- emotional trust
- follow-through

it should generally be favored.

If it increases:
- operational burden
- governance complexity
- volunteer fatigue
- architectural sophistication without operational value

it should generally be deferred.

---

# Final Acceptance Principle

Every acceptance decision should answer:

> Does this meaningfully improve the likelihood that no one is left alone?

If not:
- simplify it
- defer it
- or remove it from MVP scope.
