# USER_FLOWS.md

# Dorea Network — MVP User Flows

## Purpose

This document defines the primary user interaction flows for the Dorea MVP.

The MVP is designed to validate one core principle:

> No one is left alone.

These flows prioritize:
- simplicity
- emotional safety
- low friction
- human follow-through
- operational clarity

The MVP intentionally avoids:
- complex workflow orchestration
- advanced social mechanics
- heavy administrative interaction
- excessive configuration

---

# Primary User Roles

## 1. Member

A community participant who:
- requests support
- receives follow-up
- may encourage or support others

---

## 2. Volunteer

A trusted individual responsible for:
- acknowledgment
- follow-up
- practical coordination
- escalation support

---

## 3. Pastor / Leader

Responsible for:
- oversight
- escalation visibility
- unresolved request monitoring
- volunteer coordination

---

# Supported MVP Moments

The MVP supports only:

1. Prayer Request
2. Emotional Support
3. Practical Need
4. Escalation / Urgent Concern

---

# Flow 1 — New Member Onboarding

## Goal

Allow a new member to join quickly with minimal friction.

---

## User Steps

1. Member receives invite link or QR code
2. Member downloads or opens Dorea
3. Member creates basic profile
4. Member accepts community participation agreement
5. Member selects notification preferences
6. Member enters home screen

---

## MVP Constraints

The onboarding process should:
- complete in under 3 minutes
- require minimal typing
- avoid complex setup
- avoid advanced governance configuration

---

## Success Criteria

- onboarding completion rate is high
- users understand basic purpose
- users can request support immediately

---

# Flow 2 — Submit Prayer Request

## Goal

Allow a member to quickly request prayer.

---

## User Steps

1. Member taps “Request Support”
2. Selects “Prayer Request”
3. Enters short message
4. Optionally selects visibility level
5. Taps submit

---

## System Actions

1. Immediate acknowledgment generated
2. Request added to active queue
3. Human owner assigned
4. Follow-up reminders initiated

---

## Member Experience

The member should feel:
- seen
- acknowledged
- safe
- supported

The interaction should NOT feel:
- bureaucratic
- clinical
- administrative

---

# Flow 3 — Emotional Support Request

## Goal

Allow members experiencing discouragement, loneliness, anxiety, or emotional struggle to ask for support safely.

---

## User Steps

1. Member selects “Emotional Support”
2. Enters optional message
3. Selects urgency level
4. Submits request

---

## System Actions

1. Immediate acknowledgment delivered
2. Human owner assigned
3. Follow-up reminders activated
4. Escalation rules evaluated if necessary

---

## MVP Constraints

The MVP should:
- minimize emotional friction
- avoid overwhelming forms
- avoid requiring detailed explanations

---

# Flow 4 — Practical Need Request

## Goal

Allow members to request practical assistance.

Examples:
- meals
- transportation
- temporary help
- financial assistance coordination

---

## User Steps

1. Member selects “Practical Need”
2. Enters short description
3. Optionally selects urgency
4. Submits request

---

## System Actions

1. Request acknowledged immediately
2. Volunteer or leader assigned
3. Follow-up workflow begins
4. Resolution tracked

---

# Flow 5 — Escalation / Urgent Concern

## Goal

Support urgent human intervention when needed.

Examples:
- self-harm concern
- abuse concern
- immediate crisis
- unsafe situation

---

## User Steps

1. Member selects urgent concern
2. Enters short description
3. Submits request

---

## System Actions

1. Immediate acknowledgment generated
2. Escalation flag activated
3. Authorized leaders notified
4. Visibility restricted appropriately
5. Audit event recorded

---

## MVP Constraints

The MVP does NOT replace:
- emergency services
- licensed counseling
- law enforcement
- crisis hotlines

The system exists to:
- surface concerns
- improve visibility
- support timely human response

---

# Flow 6 — Volunteer Follow-Up

## Goal

Enable lightweight, sustainable volunteer response.

---

## Volunteer Steps

1. Volunteer receives assigned request
2. Opens request detail
3. Reviews request status
4. Contacts member
5. Logs follow-up activity
6. Marks request:
   - active
   - waiting
   - resolved
   - escalated

---

## MVP Constraints

Volunteer experience should:
- require minimal training
- avoid administrative overload
- prioritize clarity over features

---

# Flow 7 — Pastor / Leader Oversight

## Goal

Provide leadership visibility into unresolved needs.

---

## Leader Dashboard Includes

- unresolved requests
- aging requests
- escalation requests
- assignment status
- follow-up gaps

---

## Leader Actions

Leaders may:
- assign owners
- reassign requests
- escalate requests
- resolve requests
- review unresolved queues

---

## MVP Constraints

The dashboard should:
- remain simple
- prioritize visibility
- avoid excessive analytics
- avoid enterprise-style complexity

---

# Flow 8 — Request Resolution

## Goal

Ensure requests are intentionally closed.

---

## Resolution Steps

1. Volunteer or leader marks request resolved
2. Resolution timestamp recorded
3. Follow-up workflow stops
4. Request archived appropriately

---

## MVP Constraints

Requests must NOT:
- silently disappear
- auto-expire without visibility
- become hidden without accountability

---

# Flow 9 — Unresolved Request Visibility

## Goal

Prevent people from being forgotten.

---

## System Behavior

The system should surface:
- aging requests
- unassigned requests
- unresolved escalation requests
- stalled follow-up situations

---

## Operational Principle

Visibility exists to support:
- accountability
- continuity
- human follow-through

NOT:
- surveillance
- productivity scoring
- performance gamification

---

# Flow 10 — Anonymous Encouragement Feed (Minimal MVP)

## Goal

Provide lightweight encouragement without creating social pressure dynamics.

---

## Feed Behavior

The feed may include:
- anonymous encouragement
- answered prayer stories
- supportive messages
- gratitude moments

---

## MVP Constraints

The feed must avoid:
- popularity mechanics
- engagement optimization
- public competition
- influencer behavior
- performance-driven participation

---

# Human-Centered Interaction Principles

All MVP flows should prioritize:

- emotional safety
- simplicity
- acknowledgment
- continuity
- human connection
- low friction

The experience should feel:
- warm
- calm
- supportive
- trustworthy

NOT:
- corporate
- gamified
- overly technical
- workflow-heavy

---

# Deferred User Flows

The following flows are intentionally deferred beyond MVP:

- federation between churches
- advanced governance workflows
- complex multi-community routing
- advanced AI orchestration flows
- replay analytics workflows
- marketplace interactions
- advanced recommendation systems
- large-scale social interaction systems

---

# Final Flow Principle

Every user flow should answer:

> Does this interaction help ensure that no one is left alone?

If not:
- simplify it
- defer it
- or remove it from MVP scope.
