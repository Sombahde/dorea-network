# DOREA NETWORK — NON_NEGOTIABLES.md

## Purpose

This document defines the permanent architectural, operational, governance, and philosophical laws of Dorea Network.

These rules are:
- mandatory
- foundational
- non-optional

These principles protect:
- trust
- dignity
- governance integrity
- runtime consistency
- community ownership
- continuity of care

These rules supersede:
- implementation convenience
- growth pressure
- technical shortcuts
- platform optimization pressures

If implementation conflicts with this document:
- THIS DOCUMENT IS CORRECT

---

# CORE PRINCIPLE

Grace is always available.

Members are the answer to prayer.

---

# RUNTIME PRINCIPLES

## 1. Everything Is A Moment

Every capability in Dorea must execute as:
- a bounded moment
- processed through OODA
- preserved through telemetry
- versioned through sessions

There are no alternate execution models.

---

## 2. The Runtime Is The System

The runtime owns:
- orchestration
- decisions
- progression
- continuity
- telemetry
- session evolution

The frontend is only:
- a rendering surface

---

## 3. There Is Only One Runtime Authority

There must only be:
- one orchestrator
- one execution pipeline
- one session authority
- one progression model

The architecture must NEVER support:
- fragmented orchestration
- hidden execution paths
- duplicated runtime logic
- competing orchestration engines

---

## 4. Sessions Are Canonical Memory

Sessions are:
- runtime-owned
- immutable historically
- append-only
- replayable

Every completed moment creates:
- a new session version

Historical session versions must NEVER:
- be silently destroyed
- be overwritten
- lose telemetry linkage

---

## 5. Telemetry Is Foundational

Every meaningful action must generate telemetry.

Telemetry preserves:
- operational memory
- governance visibility
- replayability
- runtime explainability

If it matters:
- log it

---

## 6. Logs Are Not Operational Memory

Logs exist only for:
- debugging
- diagnostics
- provider troubleshooting

Telemetry is:
- the canonical operational history

---

# GOVERNANCE PRINCIPLES

## 7. Communities Govern Themselves

Communities govern:
- membership
- contracts
- care philosophy
- escalation behavior
- participation models

The system does NOT centrally govern:
- ministry philosophy
- pastoral behavior
- community structure

The runtime operationalizes continuity.

Communities define care.

---

## 8. Human Oversight Is Mandatory

Pastors and trusted leaders remain:
- Human-in-the-Loop oversight
- escalation authorities
- governance participants

The runtime must NEVER:
- replace pastoral discernment
- remove human governance
- operate as autonomous ministry authority

---

## 9. Communities Define Behavior Through Contracts

Behavior evolves through:
- contracts
- governance refinement
- observed moments of care

NOT through:
- uncontrolled autonomous agents
- hidden AI behavior
- ungoverned runtime mutation

The community defines the behavior.

The runtime executes the behavior.

---

# TRUST PRINCIPLES

## 10. Invitation Is Trust Propagation

Growth occurs through:
- invitation
- relationships
- community discretion
- trust propagation

The system must NEVER:
- optimize for virality
- implement growth hacking
- use engagement manipulation
- pursue acquisition funnels

Invitation is:
- portable grace
- relational extension
- trust transfer

---

## 11. Growth Is Community-Governed

Membership expansion occurs:
- at the discretion of the community
- through trusted relationships
- under governance oversight

The system must NEVER:
- pressure growth
- gamify invitations
- optimize engagement metrics
- create participation competition

---

## 12. Dignity Must Always Be Preserved

The runtime must preserve:
- privacy
- dignity
- emotional safety
- relational trust

The system must NEVER:
- publicly expose hardship
- shame members
- rank grace participation socially
- create performance dynamics around giving or need

---

## 13. Dorea Is Not Financialized

Dorea must avoid:
- banking language
- lending language
- debt language
- transactional optimization language

- grace-oriented language
- relational language
- community-centered language
- encouraging language

- “support”
- “encouragement”
- “participation”
- “grace response”

NOT:
- “loan”
- “collection”
- “borrower”
- “portfolio”

---

## 14. Members Are Never Reduced To Transactions

Members are:
- people
- participants
- neighbors
- caregivers
- recipients of grace
- providers of grace

Members are NEVER:
- revenue units
- engagement metrics
- conversion funnels
- optimization targets

---

# AI PRINCIPLES

## 15. AI Assists — It Does Not Replace Community

Dorea exists to:
- extend continuity of care
- coordinate participation
- sustain availability

AI must NEVER:
- replace prayer
- replace discernment
- replace pastoral leadership
- replace human compassion

The network responds through its members.

---

## 16. AI Must Remain Explainable

All runtime decisions must support:
- replayability
- auditability
- explainability
- telemetry reconstruction

Black-box behavioral execution is prohibited.

---

## 17. AI Must Remain Bounded

Agents must:
- operate through contracts
- execute within runtime governance
- remain telemetry-observable

Agents must NEVER:
- self-expand capabilities
- create hidden orchestration
- bypass governance
- mutate policies autonomously

---

# ARCHITECTURAL PRINCIPLES

## 18. Providers Are Utilities — Not Authorities

Providers:
- execute actions
- process communications
- perform integrations

Providers do NOT:
- own orchestration
- define policy
- control runtime flow
- mutate sessions directly

Examples:
- Stripe
- SMS gateways
- email systems

are execution utilities only.

---

## 19. Frontends Must Remain Thin

The frontend must NEVER:
- own orchestration
- manage runtime progression
- compute OODA
- mutate session history
- become the source of truth

The frontend is:
- a renderer
- a transport layer
- a relational experience

---

## 20. Historical Memory Must Remain Replayable

Moments, sessions, telemetry, and execution packages must remain:
- replayable
- reconstructable
- auditable

The runtime must support:
- historical reconstruction
- policy replay
- governance review
- operational simulation

---

- contracts
- policies
- runtime orchestration
- governed execution
