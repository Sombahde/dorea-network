# DOREA NETWORK — BUILD_PLAN.md

## Purpose

This document defines the phased implementation strategy for Dorea Network.

The purpose of this plan is to:
- stabilize architecture first
- prevent premature feature expansion
- enforce runtime consistency
- preserve governance integrity
- establish replayable operational foundations
- ensure all capabilities evolve through moments and contracts

This plan intentionally prioritizes:
- runtime foundations
- orchestration integrity
- session architecture
- telemetry
- governance

before:
- advanced UX
- feature expansion
- provider proliferation

If implementation conflicts with this document:
- THIS DOCUMENT IS CORRECT

---

# CORE BUILD PRINCIPLE

Dorea is NOT:
- a frontend-first application
- a CRUD platform
- a feature-driven SaaS product

Dorea IS:
- a governed runtime system coordinating moments of grace through OODA execution, immutable session versioning, telemetry-backed operational memory, and community-defined contracts.

The runtime must stabilize before expansion occurs.

---

# BUILD STRATEGY

The build strategy follows:

```plaintext
Foundation
  ↓
Runtime
  ↓
Moments
  ↓
Governance
  ↓
Surfaces
  ↓
Expansion
```

The runtime always comes first.
